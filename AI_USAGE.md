# AI 使用说明

## AI 提供帮助的地方

1. 辅助分析题目要求，识别核心问题是异步 HTTP 通知、可靠投递、失败处理和系统边界。
2. 辅助比较 Go、Python 以及不同基础设施方案的取舍。
3. 辅助生成第一版代码结构，包括 API、状态机、Store、Worker、Delivery 和 Retry Policy。
4. 辅助补充 README 中的系统边界、投递语义、失败策略和演进方向。
5. 辅助补充行为测试，验证重试和状态迁移。

## AI 给出但没有采纳的建议

1. 没有采用 Kafka / RabbitMQ / Redis Stream。原因：第一版不需要高吞吐消息平台，过早引入会增加部署和理解成本。当前用 `FAILED` 状态表达 MVP 阶段的逻辑 DLQ，未来吞吐和隔离需求增长后再演进为 MQ + DLQ。
2. 没有实现 exactly-once。原因：外部 HTTP API 场景下 exactly-once 需要供应商幂等或事务配合，仅靠本服务无法严格保证。
3. 没有实现完整供应商管理后台或配置 DSL。原因：题目重点是可靠投递和工程判断，后台配置不是 MVP 核心。
4. 没有实现多租户权限、审计、复杂限流和监控大盘。原因：这些属于生产化增强能力，第一版只保留能表达核心链路的功能。
5. 没有实现外部数据库或外部队列。原因：SQLite 已能在本地表达任务表、索引、事务和投递历史，不需要为 MVP 额外部署 PostgreSQL、Kafka 或 RabbitMQ。
6. 没有实现完整供应商配置中心。原因：第一版采用 HTTP envelope 降低接入复杂度，但增加了 Header 白名单、私网/metadata IP 拦截、redirect 禁止和可配置 host allowlist，避免把服务做成任意 HTTP 代理。
7. 没有解析供应商响应 body。
   原因：第一版只按 HTTP status 判断投递成功与否。响应 body 属于供应商业务语义，不同供应商差异很大，应在生产演进到供应商配置中心后再统一处理。

## 我自己做出的关键决策

1. 选择 Go。原因：我更熟悉 Go，而且本题包含 HTTP 服务、后台 worker、状态机和 IO 密集型投递，适合用 Go 实现。
2. 选择 at-least-once 投递语义。原因：这是外部 HTTP 通知场景中更现实的可靠性目标。
3. 选择先持久化再异步投递。原因：避免业务系统被外部供应商的延迟和故障拖慢。
4. 第一版使用 SQLite，而不是 JSON 文件或外部消息队列。原因：SQLite 仍然是本地嵌入式依赖，运行成本低；相比文件仓储，它能用事务和索引更准确地表达任务表、原子领取、幂等键和投递历史。
5. 限制最大重试次数。原因：供应商长期不可用时，无限重试会造成队列积压和资源浪费；进入 `FAILED` 更利于人工排查和补偿。
6. 增加投递尝试历史。原因：当前状态只能说明任务最后处于什么阶段，attempt history 可以解释每次失败的 HTTP 状态码、错误原因和重试时间，更贴近真实排障流程。
7. 保留 HTTP envelope，但补充安全边界。原因：HTTP envelope 更适合 MVP 验证可靠投递链路；同时必须限制目标地址、Header 和 redirect 行为，避免 SSRF 和敏感 Header 注入风险。
8. 暂不采用 `vendorId + eventType + payload` 作为第一版输入模型。原因：它更适合生产版本，由通知服务统一管理供应商配置、认证、模板、限流和响应解析；MVP 先用 HTTP envelope 保持实现简单，但在文档中明确其缺点是业务系统仍需了解供应商协议。
9. 将 `idempotencyKey` 定义为业务事件唯一键。
   原因：相同 key 且请求内容一致时返回已有任务，避免业务系统重试产生重复任务；相同 key 但请求内容不同时返回冲突，避免误合并不同业务事件。
