# AI 使用说明

## AI 提供帮助的地方

1. 辅助分析题目要求，识别核心问题是异步 HTTP 通知、可靠投递、失败处理和系统边界。
2. 辅助比较 Go、Python 以及不同基础设施方案的取舍。
3. 辅助生成第一版代码结构，包括 API、状态机、Store、Worker、Delivery 和 Retry Policy。
4. 辅助补充 README 中的系统边界、投递语义、失败策略和演进方向。
5. 辅助补充行为测试，验证重试和状态迁移。

## AI 给出但没有采纳的建议

1. 没有采用 Kafka / RabbitMQ / Redis Stream。  
   原因：第一版作业不需要高吞吐消息平台，过早引入会增加部署和理解成本。

2. 没有实现 exactly-once。  
   原因：外部 HTTP API 场景下 exactly-once 需要供应商幂等或事务配合，仅靠本服务无法严格保证。

3. 没有实现完整供应商管理后台或配置 DSL。  
   原因：题目重点是可靠投递和工程判断，后台配置不是 MVP 核心。

4. 没有实现多租户权限、审计、复杂限流和监控大盘。  
   原因：这些属于生产化增强能力，第一版只保留能表达核心链路的功能。

5. 没有引入第三方 Go Web 框架或数据库驱动。  
   原因：Go 标准库已经足够实现本题的 API 和 Worker，零依赖更容易在评审环境运行。

## 我自己做出的关键决策

1. 选择 Go。  
   原因：我更熟悉 Go，而且本题包含 HTTP 服务、后台 worker、状态机和 IO 密集型投递，适合用 Go 实现。

2. 选择 at-least-once 投递语义。  
   原因：这是外部 HTTP 通知场景中更现实的可靠性目标。

3. 选择先持久化再异步投递。  
   原因：避免业务系统被外部供应商的延迟和故障拖慢。

4. 第一版使用文件仓储，而不是数据库或消息队列。  
   原因：作业评估重点是设计与决策，文件仓储能表达任务持久化、状态迁移和重试语义，同时保持仓库可直接运行。生产演进时可以替换 `Store` 实现。

5. 限制最大重试次数。  
   原因：供应商长期不可用时，无限重试会造成队列积压和资源浪费；进入 `FAILED` 更利于人工排查和补偿。
